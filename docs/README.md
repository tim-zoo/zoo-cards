# 文档入口

把 `docs/` 当成这个仓库的长期知识区。项目背景、关键决策、架构约束和领域模型，后面都尽量往这里沉淀。

## 目录说明

- `project-context.md` —— 项目背景、目标、需求、时间要求和交付倾向
- `architecture/` —— 系统结构、分层方式、技术栈决策和重要技术说明
- `constraints/` —— 编码规则、项目约束、边界和不变量
- `domains/` —— 业务 / 内容领域说明

## 当前建议阅读顺序

1. `project-context.md`
2. `architecture/stack-decisions.md`
3. `architecture/overview.md`
4. `architecture/layers.md`
5. `constraints/project-constraints.md`
6. `domains/content-model.md`
7. `plans/project-plan.md`

## 使用建议

- 能写进版本库的内容，就不要只留在聊天里。
- 文档之间尽量互相链接，方便快速跳转。
- 某个决策一旦稳定下来，就把它沉淀成文档，不要反复口头解释。
- 内容模型、页面结构或约束发生变化时，同步更新对应文档。
