# 分层与依赖方向

这个项目首版采用轻量但明确的分层，目标是让内容组织、页面展示和探索路径演进都保持可解释。

## 建议的主要分层

### 1. 内容源层
职责：
- 保存原始卡片内容
- 保存路径配置、分类、标签等原始数据

典型载体：
- `content/cards/*`
- `content/paths/*`
- 其他静态配置文件

### 2. 内容装载 / 解析层
职责：
- 读取 MDX 和配置文件
- 做 schema 校验和默认值处理
- 生成稳定的领域对象

这一层负责把“松散内容文件”变成“可依赖的数据结构”。

### 3. 领域模型层
职责：
- 表达 Card、Category、Tag、Path、PathStep 等核心对象
- 约束它们之间的关系和不变量

这一层不关心页面长什么样，只关心数据结构是否合理。

### 4. 页面数据组织层
职责：
- 为首页、列表页、详情页、路径页准备所需数据
- 处理排序、分组、相关推荐、路径导航等页面级逻辑

这一层可以理解为“从领域对象到页面视图模型”的转换层。

### 5. 展示层
职责：
- 页面、布局、组件、样式、交互
- 响应式展示和品牌表达

这一层不应该直接解析原始 MDX 元数据，也不应该承载过重的内容规则。

## 允许的依赖方向

建议只允许以下方向：

`内容源层 -> 内容装载/解析层 -> 领域模型层 -> 页面数据组织层 -> 展示层`

展示层可以依赖页面数据组织层，页面数据组织层可以依赖领域模型和内容装载层，但不要反过来。

## 明确的边界要求

- 页面组件不要直接读取原始内容文件
- 展示组件不要自己猜 frontmatter 字段
- 路径页不要把路径规则散落在 UI 组件里
- 分类、标签、相关推荐逻辑应集中在数据组织层，而不是分散在多个页面中重复实现

## 需要刻意保持隔离的部分

### 内容解析 与 UI 展示
内容文件格式未来可能变化，但不应影响大部分展示组件。

### 路径逻辑 与 普通卡片展示
探索路径是建立在卡片体系之上的增强能力，不应让普通卡片浏览逻辑被路径规则绑死。

### 构建时逻辑 与 浏览器交互逻辑
静态生成、slug 计算、schema 校验等应尽量在构建时完成；浏览器端只负责消费整理好的结果。

## 当前不做的更重分层

首版不引入：
- 服务端应用层
- 数据库存储层
- 搜索服务层
- 插件式扩展系统

如果未来复杂度上升，再评估是否需要把内容服务、路径引擎或搜索能力进一步拆开。
