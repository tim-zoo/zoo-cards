# 架构概览

`zoo-cards` 是一个基于 `Next.js` 静态导出的内容驱动型网站。

首版的核心不是复杂系统，而是把 **AI 安全知识卡片** 和 **探索路径** 组织成一套清晰、可扩展、对手机和 PC 都友好的前端结构。

## 主要运行时

- **构建时**：读取本地 `MDX` 卡片内容和路径配置，生成静态页面
- **浏览器运行时**：负责页面交互、筛选、导航和响应式展示
- **部署运行时**：GitHub Pages，仅托管静态产物，不提供服务端逻辑

## 关键组成

### 1. 内容源层
项目内容以本地文件为主，至少包括：
- 知识卡片内容
- 探索路径配置
- 分类、标签、相关推荐等元数据

### 2. 内容装载与规范化层
在构建阶段读取内容文件，完成：
- frontmatter / 元数据解析
- 字段校验
- slug、分类、标签、关联关系整理
- 将原始内容转成页面可消费的稳定结构

### 3. 页面与路由层
面向用户的主要页面包括：
- 首页
- 卡片列表 / 分类浏览页
- 卡片详情页
- 路径页
- About / 项目说明页

### 4. 展示组件层
负责：
- 卡片列表和详情展示
- 路径进度和“下一步学什么”引导
- 搜索 / 筛选 UI
- 响应式布局和品牌视觉表达

## 高层数据流

1. 内容作者维护本地 MDX 和路径配置
2. 构建时读取并校验内容
3. 内容被映射成统一的领域对象和页面数据
4. 页面在构建时生成静态路由
5. 用户在浏览器中浏览、筛选、跳转和按路径探索

## 外部系统 / 依赖边界

首版只明确依赖以下外部系统：
- GitHub 仓库
- GitHub Pages 静态部署

首版**不依赖**：
- 数据库
- 服务端 API
- 用户登录系统
- 在线 CMS

## 当前架构原则

- 先做 **内容驱动**，再做复杂能力
- 先保证 **静态可部署**，再考虑运行时增强
- 先把 **卡片模型** 做稳，再叠加探索路径
- 先把 **移动端体验** 和信息层级做清楚，再增加复杂视觉效果
- 让内容组织、页面展示和路径引导保持可解释的分层
