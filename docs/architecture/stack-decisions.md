# 技术栈决策

## 当前倾向

- `Next.js` 静态导出
- `MDX` 内容驱动
- GitHub Pages 部署
- 响应式优先设计

## 设计取向

- 当前采用的主要结构：
  - 以静态站为基础的内容驱动型前端项目
  - 用卡片内容模型作为核心信息单元
  - 在卡片体系之上叠加“今天学什么”探索路径能力
- 这样选择的原因：
  - 首版目标是尽快做出一个好看、能分享、能运行的 MVP
  - GitHub Pages 静态部署最轻，不需要后端和数据库前置
  - MDX 适合知识卡片类内容表达，后续也方便扩展结构化信息块
  - 先把内容组织、响应式体验和浏览路径打磨好，比过早引入复杂系统更重要
- 当前明确不做的抽象：
  - 不先做 CMS
  - 不先做复杂知识图谱引擎
  - 不先做服务端 API 层
  - 不为未来可能的多端、多租户或个性化场景提前做过重抽象
- 后续若复杂度上升，优先演进的方向：
  - 规范内容 schema 和 frontmatter 约定
  - 明确卡片、分类、标签、路径之间的数据关系
  - 逐步拆分内容层、展示层和路径引导层
  - 视需要补充静态搜索、SEO 和分享元信息策略

## 已确认的选择

- 项目类型：`web`
- 仓库名：`zoo-cards`
- 首版主题范围：AI 安全知识卡片
- 前端技术栈：`Next.js` + 静态导出
- 内容格式：`MDX`
- 部署方式：GitHub Pages
- 仓库位置：`tim-zoo` GitHub 组织
- 视觉方向：可爱一点，但仍专业
- 首版包含：1 到 2 条“今天学什么”探索路径原型
- GitHub PR 模板：需要

## 待确认问题

- 首批卡片具体数量与主题清单
- 首版 1 到 2 条探索路径的具体走法
- 首页视觉参考、色彩方向和插画/图标策略
- GitHub Pages 的自动部署配置细节
- 首版是否需要基础 SEO / social share metadata
