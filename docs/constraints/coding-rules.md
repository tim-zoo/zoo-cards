# 编码规则

这些规则是为这个项目首版量身定的，重点是保持静态站项目的清晰度、可维护性和后续 agent 接手成本。

## 结构规则

- 页面组件优先负责组合，不负责解析原始内容数据
- 内容解析、schema 校验、slug 处理等逻辑集中在内容装载层
- 路径推荐、相关推荐、分组排序等规则尽量集中，不要散落在多个页面重复实现
- 文件职责保持聚焦，避免“一个文件同时做内容、路由、样式、业务规则”

## 类型与数据规则

- 优先显式类型，而不是靠组件内部猜字段
- 对 Card、Path、PathStep 等核心对象保持稳定字段约定
- 所有跨文件引用优先使用稳定 slug，而不是标题文本
- 内容层允许作者友好，但进入页面前必须转换成程序友好的结构

## UI 与样式规则

- 先做移动端可用，再增强 PC 布局
- 动物园风格只作为品牌表达，不牺牲信息密度和可读性
- 组件应优先可复用，避免为单一页面堆砌大量特例样式

## 变更规则

- 改内容模型时，同步更新相关文档
- 改页面结构时，考虑是否影响路径、相关推荐和响应式布局
- 引入新约束时，优先沉淀到文档和验证入口，而不是只留在口头说明里

## 验证规则

- 行为变更后，至少补充对应的构建或验证手段
- 能在静态构建阶段发现的问题，不要拖到运行时
- `scripts/validate.sh` 应逐步收敛成统一入口，避免验证命令散落
