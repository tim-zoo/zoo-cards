## 背景

项目仓库已经完成初始化，产品方向、技术栈和约束已基本明确。下一步需要把“文档里的项目”变成“可运行的项目”，先搭建一个适合 GitHub Pages 静态部署的应用骨架。

这个阶段的关键不是堆功能，而是打通以下基础：
- Next.js 静态导出链路
- 基础内容目录结构
- 卡片 / 路径的最小数据模型落地
- 首页、卡片页、路径页的最小可运行壳子
- 统一验证入口

## 目标

搭建首版应用代码骨架，使项目具备以下能力：
- 本地可运行
- 可静态导出
- 具备清晰的内容目录结构
- 有最小但真实的页面与示例内容
- 后续能在此基础上按计划逐步补功能和视觉

## 非目标

- 搜索服务、数据库、登录系统等扩展能力

## 设计思路

### 1. 先把静态站链路打通
选择 `Next.js` 静态导出，确保从一开始就围绕 GitHub Pages 部署目标设计，而不是后面再迁。

### 2. 先落内容模型，再做页面细节
先定义卡片和路径的文件结构、字段和 loader 约定，避免页面写起来后反复返工。

### 3. 页面先做最小壳子
首页、卡片详情页、路径页优先做“有真实数据流的壳子”，而不是直接追求最终视觉完成度。

### 4. 验证入口尽早收敛
尽早让 `scripts/validate.sh` 变成真实命令入口，哪怕首版只包含安装检查、lint、build 或 export。

## 实施步骤

### Step 1，初始化前端工程
- 建立 `Next.js` 项目基础结构
- 配置 TypeScript、静态资源和基础目录
- 确认适合 GitHub Pages 的静态导出配置

### Step 2，建立内容目录与最小 schema 约定
- 创建卡片内容目录
- 创建路径配置目录
- 约定 frontmatter / 元数据字段
- 准备最小示例卡片和示例路径

### Step 3，建立内容装载与数据组织骨架
- 创建内容 loader / parser 的基础模块
- 输出最小 Card / Path 数据结构

### Step 4，搭建最小页面壳子

- 卡片列表 / 分类浏览页：最小数据展示

### Step 5，建立基础验证入口
- 让 `scripts/validate.sh` 调用真实命令
- 至少覆盖：安装后可构建、可静态导出、关键页面无明显结构错误

## 风险与回滚

### 主要风险
- GitHub Pages 静态导出配置细节踩坑
- 内容模型过早复杂化
- 页面壳子一开始写得太重，反而拖慢验证节奏

### 控制方式
- 优先保住静态导出和样例数据链路
- 用最小真实数据打通流程，而不是先铺大量空页面
- 遇到框架或部署细节复杂时，优先选更轻、更稳的实现

### 回滚思路
- 若某种内容组织方式过重，回退到更简单的 MDX + 配置方案
- 若静态导出方案出现不必要复杂度，优先调整目录和配置，不贸然引入后端依赖
