# 项目宏观计划

## 项目目标

做一个基于 GitHub Pages 部署的响应式知识卡片网站，首版以“动物园风格知识卡片站”为核心形态，围绕 AI 安全主题提供可浏览、可搜索、可分享的卡片体验，并在卡片体系之上提供 1 到 2 条“今天学什么”探索路径原型，为后续长期扩展打下轻量但清晰的结构基础。

## 当前阶段判断

当前处于项目初始化和 MVP 规划阶段。重点不是做重架构，而是先把仓库骨架、项目文档、内容组织思路和第一阶段范围明确下来。

## 阶段划分

### Phase 1

- 目标：完成仓库初始化，明确内容模型、页面信息架构和首版范围
- 预期交付：项目骨架、基础文档、首版 MVP 边界、可执行的后续实施方向
- 当前先不做：复杂后端、账号系统、数据库、自动生成路径、复杂图谱

### Phase 2

- 目标：完成首版 MVP 开发并可部署到 GitHub Pages
- 预期交付：首页、卡片列表/分类浏览、卡片详情、搜索或筛选、相关推荐、1 到 2 条探索路径原型、基础响应式体验，以及首期轻游戏化学习闭环
- 主要风险：内容模型不够清晰、视觉风格摇摆、首批卡片数量和质量不足、Pages 部署细节未提前打通、游戏化机制打断阅读节奏

### Phase 3

- 目标：在首版可用基础上逐步增强内容规模、探索体验和品牌表达
- 预期交付：更多卡片主题、更成熟的路径引导、可能的 SEO/share 优化、内容 schema 演进，以及更完整的学习进度与成就体系
- 主要风险：功能膨胀过快、内容维护成本上升、结构在增长后失去清晰度、奖励机制缺少真实价值导致用户疲劳

## 当前优先级

当前最优先推进的是：
- 明确首批卡片内容范围
- 明确 1 到 2 条探索路径原型
- 选定页面结构和首版视觉方向
- 搭起实际前端代码骨架，并尽早打通本地构建和静态部署链路
- 为泛用户补齐轻游戏化学习闭环，包括答题、即时反馈、奖励和继续学习引导

## 关键约束

- 必须适合 GitHub Pages 静态部署
- 必须优先保证手机和 PC 的响应式体验
- 首版不依赖数据库和服务端 API
- 首版要轻量，不做过度设计
- 风格可以可爱，但要保持专业感
- 游戏化设计要服务学习，不应把产品做成重考试或重游戏

## 时间要求 / 里程碑

- 当前没有强制正式上线时间
- 当前策略是先尽快做出一个好看、能分享、能运行的 MVP
- 建议尽早形成第一个可本地运行和静态导出的可视版本，再逐步补内容和路径细节
- 在 MVP 阶段中，游戏化能力优先做到“可感知、可验证、可迭代”，而不是一次做满

## 什么时候拆成执行计划

当某个阶段准备进入具体实现，并且任务复杂度已经需要多文件修改、分阶段推进或分阶段验证时，再由 code agent 和用户确认后，在 `plans/active/` 中创建具体执行计划。
